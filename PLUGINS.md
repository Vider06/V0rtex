# V0RTEX Plugin System

> **Status:** The plugin loader is currently under active development and not yet fully implemented. This document defines the rules and structure that will govern the plugin system once released. Check the repository for updates.

---

## What is a Plugin?

A plugin is an external Python script that extends V0RTEX with additional functionality. Plugins are loaded through the dedicated tab in **CFG → PLUGINS** and run inside V0RTEX without modifying the main application in any way.

Plugins are a way to add features, integrations, or custom analysis tools on top of V0RTEX — not a way to patch or alter it.

The official plugin repository is maintained separately:
> 🔗 **[[V0RTEX Plugin Hub](https://github.com/Vider06/V0rtex-Plugin-Official)](https://github.com/Vider06/V0rtex-Plugin-Official)** *(link will be updated when the repository is created)*

## Plugin Classes

Classes are listed from highest to lowest trust level.

### 🔴 V0RTEX-Made
V0RTEX-Made plugins are written exclusively by Vider_06. They are used to ship beta features, alpha experiments, optional functionality, or capabilities that are not yet part of the official V0RTEX release but may be useful to advanced users.

- Written and signed by Vider_06 only — no exceptions
- Have complete, unrestricted communication with V0RTEX internals
- Have the same permission level as V0RTEX itself
- May access any file, thread, internal state, scan result, UI element, or configuration
- May be updated or removed at any time without notice
- Distributed exclusively through the official V0RTEX plugin repository

**V0RTEX-Made plugins are not a mechanism for the community.** They are an internal tool for the Author. No community plugin can ever reach this class.

> ⛔ **Account Ban Warning**
>
> Attempting to impersonate a V0RTEX-Made plugin — by forging its signature, injecting code to fake its class, manipulating the plugin header, bypassing the class verification system in any way, or distributing a plugin that falsely claims to be V0RTEX-Made, Elevated, or Verified — results in a **permanent account ban from the V0RTEX plugin ecosystem**. This means your GitHub account will be blocked from submitting, publishing, or having any plugin listed in the official repository, permanently and without appeal. V0RTEX will also flag and block all plugins associated with your account.
>
> This is the most severe violation possible in the V0RTEX plugin system. There are no exceptions and no second chances.

### ⚡ Elevated
Elevated plugins are Verified plugins that have been granted special permissions by the Author (Vider_06) beyond the standard ruleset. These permissions are explicitly listed in the plugin's entry in the official repository and displayed to the user before loading.

Elevated status is granted exclusively by Vider_06 on a case-by-case basis. A plugin cannot request or self-declare Elevated status — it must go through the standard Verified validation process first, and then the Author may grant Elevated permissions if the use case justifies it.

V0RTEX will display the full list of granted special permissions to the user before loading an Elevated plugin. The user must explicitly confirm before the plugin is allowed to run.

#### Elevated Permission Catalogue

| Permission | Description |
|---|---|
| `fs.read.external` | Read files and folders outside the `V0rtex_System/` directory. **High risk** — user is warned explicitly. |
| `fs.write.zip` | Create `.zip` archive files in the plugin's own subfolder |
| `fs.write.json` | Create and write `.json` files in the plugin's own subfolder |
| `fs.write.xml` | Create and write `.xml` files in the plugin's own subfolder |
| `fs.write.csv` | Create and write `.csv` files in the plugin's own subfolder |
| `fs.write.html` | Create and write `.html` files in the plugin's own subfolder |
| `fs.write.pdf` | Create and write `.pdf` files in the plugin's own subfolder |
| `fs.read.v0rtex.logs` | Read V0RTEX debug and session log files from `debug_log/` |
| `fs.read.v0rtex.reports` | Read scan reports generated by V0RTEX from `reports/` |
| `net.background` | Make outbound network requests in a background thread (must be disclosed in header) |
| `net.listen` | Open a local network listener or socket |
| `proc.read` | Read the list of running processes on the system |
| `sys.read.registry` | Read Windows registry keys |
| `sys.read.env` | Read system environment variables |

### ✅ Verified

Verified plugins are reviewed, approved, and digitally signed by the D.V.V. (Dipartimento di Verifica V0RTEX). They are published in the official plugin repository and have access to a controlled, read-only V0RTEX API surface.

#### Verified V0RTEX API Access

| API | Description |
|---|---|
| `vx.ui.add_tab(name, frame)` | Register a new tab in the V0RTEX main tab bar |
| `vx.ui.add_menu_entry(label, callback)` | Add an entry to the plugin menu section |
| `vx.ui.notify(message, level)` | Display a notification in the V0RTEX status bar (`info`, `warning`, `error`) |
| `vx.ui.get_theme()` | Read current UI theme colors and font sizes |
| `vx.scan.get_last_result()` | Read the most recent scan result — read only |
| `vx.scan.on_scan_complete(callback)` | Register a callback called after each scan |
| `vx.scan.get_history(limit)` | Read the last N scan results — read only |
| `vx.ioc.get_last_extracted()` | Read the IOC list from the most recently analyzed file |
| `vx.notes.append(text)` | Append a line to the V0RTEX notes scratchpad |
| `vx.version.get()` | Read the current V0RTEX version string |

All Verified API calls are read-only or additive. Any call outside this list raises a `PluginPermissionError`.

---

### ⚠️ Unverified
Unverified plugins have not been reviewed or signed. V0RTEX displays a warning before loading them. No V0RTEX API access. Use at your own risk.

---

### 🚫 Banned
Banned plugins are permanently blocked. V0RTEX checks every plugin against the banned list on every startup. A banned plugin cannot be loaded under any circumstances — there is no override, no button, no setting that changes this.

## Installation

Plugins must be placed in:
```
V0rtex_System\V0RTEX_v<version>\plugins\
```

Once placed in the correct folder, the plugin appears in **CFG → PLUGINS**, where it can be loaded, enabled, or disabled.

The easiest way to install plugins is through the **V0RTEX Plugin Manager** — a standalone tool available in the official plugin repository that lets you browse, install, update, and uninstall plugins directly.

---

## Pre-Launch Security Checks

Every time V0RTEX starts, before any plugin is loaded, it runs a full automated check on every plugin in the `plugins/` folder. This cannot be skipped or disabled.

| Check | Description |
|---|---|
| **Malware scan** | Scanned with V0RTEX's own engine — entropy, YARA, suspicious strings, known signatures |
| **Hash verification** | SHA-256 compared against the official repository record. Mismatch = blocked for Verified and above |
| **Repository status** | Current class and status fetched from the plugin repository. Newly banned plugins are caught here |
| **Class consistency** | Header class vs official record. Mismatch = tampered, blocked |
| **Signature check** | Digital signature verified for Verified, Elevated, V0RTEX-Made. Invalid = blocked |
| **Banned list** | Checked against the banned list. Match = blocked, no override |
| **Dependency audit** | Declared imports vs actual imports. Undeclared = warning (Unverified) or block (Verified+) |

Offline mode skips internet-dependent checks gracefully but never skips local scan, signature, or cached banned list.

## Automatic Updates

> **Planned — not yet implemented.**

Verified, Elevated, and V0RTEX-Made plugins will support automatic updates from the official repository.

- V0RTEX checks for updates on startup
- Updates are downloaded, hash and signature verified, old version backed up
- Elevated plugins require re-confirmation if permissions change
- Unverified plugins are never auto-updated
- Locally modified plugins are never auto-updated

## Plugin Update Process

When a Verified or Elevated plugin has a new version ready, the developer must submit an **Update Request** through the official plugin repository. See the Plugin Hub for instructions.

The D.V.V. (Dipartimento di Verifica V0RTEX) re-reviews the plugin from scratch for every update. Updates can be accepted or rejected. See the Plugin Hub documentation for the full update process and cooldown rules.

## Rules for Plugin Developers

### ✅ You MAY

- Add new UI elements as separate Tkinter windows — may remain open while V0RTEX is running
- Read files provided by the user through your own interface
- Make outbound network requests when triggered by the user
- Make background network requests only if granted `net.background` and declared in header
- Use any Python standard library module
- Use third-party packages declared in the header
- Write `.txt` or `.md` files in your own plugin subfolder (or extra formats if Elevated)
- Use the Verified V0RTEX API if Verified or above
- Communicate with other plugins passively through shared files

### ❌ You MAY NOT

- Import, read, modify, or interact with `v0rtex.py` in any way
- Access V0RTEX internal files without explicit Elevated permission
- Inject code or monkey-patches into V0RTEX
- Call V0RTEX API methods outside your permitted list
- Monitor or interact with V0RTEX's internal threads
- Write files outside your subfolder
- Create forbidden file formats
- Execute or import any file you created
- Activate, deactivate, or trigger another plugin
- Import another plugin's module directly
- Spawn persistent background threads without `net.background`
- Attempt privilege escalation
- Forge, spoof, or fake your plugin's class or signature
- Distribute a plugin claiming false class status
- Distribute a repackaged version of V0RTEX

## Background Network Activity

Requires `net.background` (Elevated). Undeclared background network activity in any plugin is a Level 2 violation minimum.

Rules when granted:
- Header must list all contacted domains in `Background-Endpoints`
- User must be informed when background activity starts and stops
- Threads must terminate cleanly on unload
- No user/system data may be transmitted without explicit user action

---

## Inter-Plugin Communication

Passive, file-based only. Plugins may read `.txt` / `.md` files written by other plugins from their subfolders. Direct module imports, function calls, or triggering another plugin's `run()` are strictly forbidden.

---

## File Output Rules

| Rule | Detail |
|---|---|
| Default formats | `.txt`, `.md` |
| Elevated formats | As per granted permissions |
| Location | Plugin's own subfolder only |
| Always forbidden | `.py`, `.exe`, `.bat`, `.sh`, `.ps1`, `.vbs`, `.js` and all executables |
| Execution | Output files must never be executed or imported |

## Plugin Structure

### Required Header

```python
# V0RTEX Plugin
# Name: My Plugin Name
# Version: 1.0.0
# Author: YourName
# Description: One-line description
# Dependencies: requests, beautifulsoup4   (or "none")
# Class: Unverified
# Elevated-Permissions: none
# Background-Network: no
# Background-Endpoints: none
# Min-V0RTEX-Version: 1.0.0
```

### Required Entry Point

```python
def run():
    pass
```

### Optional Lifecycle Hooks

```python
def on_load():    pass   # called once on load — init, register callbacks
def on_unload():  pass   # called on disable/close — MUST clean up threads and files
def on_update(old_version, new_version): pass  # called after auto-update
```

`on_unload()` is strongly recommended for any plugin that opens files or spawns threads.

## Violations and Enforcement

### Ban Levels

#### 🟡 Level 1 — Warning
Plugin remains loadable. Warning badge in CFG → PLUGINS. Developer notified on the plugin's Discussion thread. Fix required within **30 days** or escalates to Level 2.

Triggers: minor header inaccuracy, unused dependency, missing `on_unload()` (first notice), wrong `Min-V0RTEX-Version`, misleading description.

#### 🟠 Level 2 — Plugin Suspension
Plugin immediately suspended and blocked. Developer has **14 days** to fix and re-submit. One appeal allowed. Expiry = automatic Level 3.

Triggers: undeclared dependency, forbidden file type (first), undisclosed background network, incomplete `Background-Endpoints`, persistent thread without permission, unpermitted API call, Level 1 ignored.

#### 🔴 Level 3 — Permanent Plugin Ban
Plugin permanently added to banned list. Blocked on every machine forever. Developer may still submit other plugins but receives heightened scrutiny. Appeal within **7 days** only.

Triggers: confirmed malicious code, data exfiltration, accessing V0RTEX internals, obfuscated code, bad-faith fix submission, plugin resubmitted under different name, three Level 2s on same plugin.

#### ⛔ Level 4 — Permanent Account Ban
GitHub account permanently banned from the entire V0RTEX plugin ecosystem. All associated plugins banned. Account listed publicly. No appeal. Ever.

Triggers: forging signature, impersonating Verified/Elevated/V0RTEX-Made, bypassing class verification, distributing plugin designed to harm users, backdoor/RAT/ransomware, attacking repository infrastructure, ban evasion.

### Escalation Path

```
🟡 Level 1 → fixed ✅ / ignored 30d → 🟠 Level 2
🟠 Level 2 → fixed ✅ / expired 14d → 🔴 Level 3
🔴 Level 3 → severe/evasion → ⛔ Level 4
⛔ Level 4 → no path back
```
